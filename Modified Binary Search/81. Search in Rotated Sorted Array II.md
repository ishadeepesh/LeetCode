# 🔎 81. Search in Rotated Sorted Array II

## 🧩 Problem

You are given an integer array `nums` sorted in **non-decreasing order**.

Before reaching your function, the array may have been **rotated at an unknown pivot**.

### Example

Original array

```python
[0,1,2,4,4,4,5,6,6,7]
```

After rotation (pivot = 5)

```python
[4,5,6,6,7,0,1,2,4,4]
```

⚠️ Unlike the previous problem, **duplicates are allowed**.

🎯 **Goal**

Return:

* `True` → if `target` exists in the array
* `False` → otherwise

You should try to **minimize the number of operations**.

---

# 💡 Key Idea

Normally in **rotated binary search**, we detect which half is sorted.

Example:

```
[4,5,6,7 | 0,1,2]
```

But duplicates can create an **ambiguous situation**:

```
[1,0,1,1,1]
 lo mid   hi
```

Here:

```
nums[lo] == nums[mid] == nums[hi]
```

Now we **cannot determine which half is sorted**.

### 🚨 Solution

Shrink the search window:

```
lo += 1
hi -= 1
```

This removes duplicates from the boundary and allows the algorithm to continue.

---

# 🧠 Strategy

During each step of binary search:

1️⃣ Compute `mid`
2️⃣ If `nums[mid] == target` → return `True`
3️⃣ If duplicates cause ambiguity
  → shrink search space (`lo++`, `hi--`)
4️⃣ Otherwise determine the **sorted half**
5️⃣ Continue searching in the appropriate half

---

# 💻 Python Implementation

```python
class Solution:
    def search(self, nums: List[int], target: int) -> bool:

        lo, hi = 0, len(nums) - 1

        while lo <= hi:

            mid = (lo + hi) // 2

            # 🎯 Target found
            if nums[mid] == target:
                return True

            # ⚠️ Ambiguous case due to duplicates
            if nums[lo] == nums[mid] == nums[hi]:
                lo += 1
                hi -= 1

            # 📈 Left half is sorted
            elif nums[lo] <= nums[mid]:
                if nums[lo] <= target < nums[mid]:
                    hi = mid - 1
                else:
                    lo = mid + 1

            # 📈 Right half is sorted
            else:
                if nums[mid] < target <= nums[hi]:
                    lo = mid + 1
                else:
                    hi = mid - 1

        return False
```

---

# 📌 Example

**Input**

```python
nums = [2,5,6,0,0,1,2]
target = 0
```

**Output**

```
True
```

📍 Because `0` exists in the array.

# 📊 Complexity Analysis

| Type         | Complexity   |
| ------------ | ------------ |
| ⏱ Best Case  | **O(log n)** |
| ⏱ Worst Case | **O(n)**     |
| 📦 Space     | **O(1)**     |

Worst case happens when the array contains **many duplicates**.

Example:

```
[1,1,1,1,1,1,1]
```

Binary search cannot eliminate half the array efficiently.

---

# ⚠️ Edge Cases

✔ Target not present
✔ Array contains many duplicates
✔ Array not rotated
✔ Single element array

---

# 🔑 Key Difference from Problem 33

Problem 33 (without duplicates):

```
Binary search always determines a sorted half
```

Problem 81 (with duplicates):

```
Duplicates may hide the sorted half
```

So we add one extra rule:

```
if nums[lo] == nums[mid] == nums[hi]:
    lo += 1
    hi -= 1
```

---

# 🏁 Final Takeaway

This problem tests whether you can **adapt binary search when duplicates break the normal sorted-half detection**.

The trick is simple but powerful:

> 🚪 When duplicates create ambiguity, **shrink the search window**.

Once the ambiguity disappears, normal rotated binary search continues.

🚀 A classic interview problem combining **binary search + edge case handling**.
